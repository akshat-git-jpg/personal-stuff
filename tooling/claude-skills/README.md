# claude-skills — single source of truth for custom Claude Code skills

All hand-authored Claude Code skills live here, version-controlled. Each Claude
account loads them via **symlinks** that `scripts/relink.sh` (at the repo root)
creates into the account's config dir. There is one physical copy (here) — so
editing a skill here, or having Claude edit it in a session, is the same file.
No sync, no drift.

## Layout

```
tooling/claude-skills/
  <skill-name>/SKILL.md   ... skill folders (the real source)
  manifest/work.txt       ... which skills the WORK account gets
  manifest/personal.txt   ... which skills the PERSONAL account gets
```
The linker itself lives at `scripts/relink.sh` (it resolves this store relative
to its own location, so it survives a repo move/rename).

## The work / personal / both distinction

This is enforced by the two manifest files — one skill name per line:

- name in **both** files  → shared (linked into both accounts)
- name in **work.txt only**     → work-only
- name in **personal.txt only** → personal-only

Current split: work-only = `create-feature-plan`, `use-zluri-docs`,
`zluri-release`; personal-only = `valyu-best-practices`; everything else shared.

## How skills load (confirmed behavior)

Claude scans `<config>/skills/` at session start and follows symlinks, so a
symlinked skill is indistinguishable from a real one: it appears in `/skills`,
the slash/dropdown menu (needs `user-invocable: true`), and auto-invokes on its
description — identical to a local skill.

## Common tasks

**Add a new skill (shared):**
```bash
mkdir -p claude-skills/my-skill && $EDITOR claude-skills/my-skill/SKILL.md
echo my-skill >> tooling/claude-skills/manifest/work.txt
echo my-skill >> tooling/claude-skills/manifest/personal.txt
./scripts/relink.sh
# then RESTART the claude-work / claude-personal session (discovery is cached)
```
For work-only or personal-only, add the line to just one manifest.

**Move a skill between accounts:** edit the manifest lines, rerun `relink.sh`.

**New laptop:** clone this repo → `./scripts/relink.sh` → recreate the
`claude-work`/`claude-personal` zsh aliases → re-`/plugin install` the plugins
(`superpowers`, `frontend-design`) per account. For `pinterest-research`, run
`npm install` inside its folder (node_modules is gitignored).

## Not managed here

The `pp-elevenlabs`, `pp-linear`, `pp-openrouter`, `pp-sentry` skills are
generated by the printing-press flow and symlinked from `~/.agents/skills/`.
They stay work-only and are left untouched by `relink.sh`.

Plugins (`superpowers`, `frontend-design`) are per-account installs and can't
be symlinked — reinstall them with `/plugin install` on a new machine.
