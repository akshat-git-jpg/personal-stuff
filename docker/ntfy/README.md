# ntfy (self-hosted)

Push-notification server backing [pp-ntfy](../../cli/ntfy/README.md). Runs
on the Hostinger VPS at `http://srv1377177.hstgr.cloud:8888`.

## Why self-hosted (and why HTTP, not HTTPS — for now)

- Privacy: alarm payloads never touch the public `ntfy.sh` instance.
- Cost: $0 — fits inside the existing Hostinger VPS.
- HTTP-only is a temporary state. Let's Encrypt has a per-registered-domain
  rate limit (50000 certs/week) on `hstgr.cloud` shared across all
  Hostinger free subdomains, which was saturated at deploy time.
  See [Upgrading to HTTPS](#upgrading-to-https) below.

Security model while on HTTP: **topic name = the secret.** ntfy is
configured with `auth-default-access: read-write`, but topic names are
high-entropy random strings (generated by `openssl rand -hex 12`), so
they aren't enumerable. This matches the hosted ntfy.sh threat model.
For a stronger guarantee (auth required to publish or subscribe), see
[Adding auth](#adding-auth) below.

## VPS layout

```
/docker/ntfy/
├── docker-compose.yml
└── data/
    ├── cache/                ← SQLite message cache (gitignored)
    │   └── cache.db
    └── etc/
        └── server.yml        ← ntfy config
```

Reachable on:
- VPS-local: `http://localhost:8888`
- Public: `http://srv1377177.hstgr.cloud:8888`  (Hostinger firewall rule 1040683 allows `TCP 8888 from any`)

## Files in this repo

| File | Purpose |
|------|---------|
| `docker-compose.yml` | The compose stack (mirror of `/docker/ntfy/docker-compose.yml` on VPS) |
| `data/etc/server.yml` | ntfy config (mirror of `/docker/ntfy/data/etc/server.yml`) |
| `README.md` | This file |
| `data/cache/` | Gitignored — runtime message cache only |

Note: these files are kept in git for version control, but they are NOT
auto-deployed on VPS git pull. Changes must be manually applied:

```bash
ssh hostinger-vps
cp /srv/projects/personal-stuff/docker/ntfy/docker-compose.yml /docker/ntfy/
cp /srv/projects/personal-stuff/docker/ntfy/data/etc/server.yml /docker/ntfy/data/etc/
cd /docker/ntfy && docker compose up -d
```

## Upgrading to HTTPS

Once Let's Encrypt's `hstgr.cloud` quota resets (typically days), we can
move ntfy behind the existing Traefik stack at `/docker/n8n/`:

1. Stop ntfy: `cd /docker/ntfy && docker compose down`
2. Restore the Traefik-labeled version of `docker-compose.yml` (see git
   history of this file — the pre-rate-limit revision).
3. Set `base-url: "https://ntfy.srv1377177.hstgr.cloud"` in `server.yml`,
   `behind-proxy: true`, `listen-http: ":80"` (Traefik handles TLS termination).
4. `docker compose up -d` and watch `docker logs n8n-traefik-1` for the
   ACME success line.
5. Once `https://ntfy.srv1377177.hstgr.cloud/v1/health` returns 200, close
   the 8888 firewall rule via Hostinger MCP / panel.
6. Update `cli/ntfy/pp_ntfy.py` `DEFAULT_SERVER` to the HTTPS URL.

## Adding auth

If topic-as-secret feels weak (e.g., you want to grant readonly access to
someone without giving them publish ability):

```bash
# On VPS
docker exec -it ntfy ntfy user add --role=admin akshat
# enter password when prompted

# Then edit /docker/ntfy/data/etc/server.yml:
#   auth-default-access: "deny-all"
#   auth-file: "/var/cache/ntfy/users.db"

# Restart
cd /docker/ntfy && docker compose restart
```

Then either:
- Embed user:pass in the URL: `http://akshat:pass@srv1377177.hstgr.cloud:8888/topic`
- Send `Authorization: Bearer <token>` header (better; ntfy supports token-based auth)

The phone app has a per-server credentials field — populate it once.
